# 执行清单

基于当前项目阶段，任务按 `P0 / P1 / P2` 排序。原则是先建立质量闭环，再收紧工程边界，最后升级检索能力。

## P0 质量闭环

### 评测集

- [ ] 新建评测数据目录，例如 `data/eval/`
- [ ] 整理 30 到 50 个真实问题样本
- [ ] 覆盖事实问答、步骤问答、拒答场景
- [ ] 为每个问题补充期望答案或关键事实点
- [ ] 为每个问题标注期望命中的文档来源

### 检索评测

- [ ] 新建 `app/eval/retrieval_eval.py`
- [ ] 实现单问题检索评测入口
- [ ] 输出 top-k 命中情况
- [ ] 统计命中率、召回率或简单 hit rate
- [ ] 支持对比不同 `search_type`、`top_k`、`fetch_k`

### 回答评测

- [ ] 新建 `app/eval/answer_eval.py`
- [ ] 建立回答结果采样脚本
- [ ] 对回答做人工评分或规则评分
- [ ] 至少区分正确、部分正确、无法回答但合理、幻觉四类结果
- [ ] 沉淀 bad case 样本

### 可观测

- [ ] 给一次完整问答记录 query、retrieved chunks、answer、citations
- [ ] 明确日志字段，便于后续回放问题
- [ ] 预留 traces 接口位置，后续可接 LangSmith 或其他 tracing

## P1 工程边界

### 服务层

- [ ] 新建 `app/services/rag_service.py`
- [ ] 把检索与回答主流程收敛到 service 层
- [ ] 让 CLI、Streamlit 调同一 service 入口
- [ ] 避免 UI 层直接拼装业务流程

### Agent 与 Middleware

- [ ] 明确 `create_agent.py` 只负责模型、工具、middleware 装配
- [ ] 把当前 middleware 真正接入主链路
- [ ] 梳理 prompt 与动态上下文注入策略
- [ ] 明确哪些逻辑属于 tool，哪些逻辑属于 agent

### 配置与基础设施

- [ ] 给 `Settings` 增加基础校验
- [ ] 明确必填环境变量和默认值策略
- [ ] 缓存 embeddings 实例
- [ ] 缓存 vector store 实例
- [ ] 统一入口文件，减少重复启动路径

### 文档结构

- [ ] 保持 `README` 简洁，只做导航和项目介绍
- [ ] 现状、路线图、执行清单放在 `docs/`
- [ ] 后续新增架构说明时单独放文档，不继续堆进 `README`

## P2 检索升级

### 召回质量

- [ ] 引入 reranker
- [ ] 增加 metadata 过滤能力
- [ ] 评估是否需要 hybrid search
- [ ] 对不同文档类型调整 chunk 策略

### 上下文质量

- [ ] 抽离 formatter 模块
- [ ] 增加上下文裁剪与去重策略
- [ ] 控制传给模型的上下文长度
- [ ] 优化 citation 输出结构

### 入库能力

- [ ] 抽离 loaders 模块
- [ ] 支持更多文档类型，例如 docx、html
- [ ] 明确增量入库与重建索引策略
- [ ] 为入库过程补充更细粒度日志

## 中期任务

- [ ] 增加 API 层
- [ ] 持久化会话状态，替换 `InMemorySaver`
- [ ] 增加文档删除与重建能力
- [ ] 增加用户反馈闭环
- [ ] 增加基础运行指标监控

## 当前不做

下面这些事情当前不建议优先推进:

- [ ] 同时支持很多向量库
- [ ] 过早做复杂 memory
- [ ] 在没有评测前频繁调 prompt
- [ ] 一次性把理想目录全部铺开
- [ ] 在主链路还没稳定前增加太多 tool

## 建议执行顺序

1. 先完成 `P0` 中的评测集和基础评测脚本
2. 再完成 `P1` 中的 `rag_service` 和 middleware 接入
3. 然后推进 `P2` 中的 reranker 和上下文压缩
4. 最后进入 API、持久化、反馈闭环这些中期任务
